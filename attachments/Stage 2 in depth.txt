What a Random Forest Actually Is
A Random Forest is a collection of decision trees — you're correct. But the key detail is why multiple trees instead of one.

A single decision tree is brittle. It memorizes the training data too well — you show it slightly different traffic and it gets confused. This is called overfitting.

The Random Forest fixes this by building many trees, each trained on a random subset of the training data and a random subset of the features. No two trees see exactly the same picture. Each tree is deliberately limited so it can't overfit the whole dataset.

Then when a new feature vector arrives, every tree independently votes — attack or benign. The forest counts the votes and produces a probability: if 97 out of 100 trees say attack, P = 0.97. If 45 say attack, P = 0.45.

That probability is what Stage 2 acts on.

Attack subsides → k decrements each quiet window
k hits zero     → IP removed from ipset

## Stage 2 Full Workflow

### Layer 1 — IPC Input

Python opens `/tmp/stage1.sock` and listens for incoming bytes. When Stage 1 triggers, it writes the raw bytes of the `#[repr(C)]` feature struct into the socket. Python reads exactly the right number of bytes and calls `struct.unpack` with a format string that matches the Rust memory layout exactly — 7 `f64` values plus a timestamp. This produces 8 Python floats ready for classification.

One critical detail: Python's `struct.unpack` format string must match the Rust struct field order and sizes **exactly**. If Rust sends `h, r, mean_h, mean_r, sigma_h, sigma_r, proto_ratio, timestamp` in that order, Python must unpack in that exact order. A single field swap is a silent bug — the numbers unpack without error but mean completely different things to the RF.

### Layer 2 — Classification

The 7 features (excluding timestamp) are assembled into a numpy array and passed to the trained Random Forest via `predict_proba()`. This returns the probability P that the traffic is an attack. The three-way decision is made here.

One important thing to understand: `predict_proba()` returns two values — `[P_benign, P_attack]`. They sum to 1.0. You use `P_attack` (the second value) for your threshold comparisons.

### Layer 3 — Action

Three possible outcomes:

**P > 0.6 — Block:**
Extract the source IP from the feature context, run `ipset add blocklist <ip>`, increment its `block_count`, set `k_remaining` based on `block_count`. The single `iptables` rule already in place drops everything matching the `blocklist` set — no further action needed per packet.

**0.4 ≤ P ≤ 0.6 — Monitor:**
Log the event with timestamp and feature values. Watch subsequent windows. Don't block yet.

**P < 0.4 — Benign:**
Do nothing. Log if you want observability.

### k-Decay Loop

Runs on every incoming window event regardless of classification outcome. For every IP in `blocked_ips`:
- If Stage 1 is still flagging anomalies involving that IP — reset k
- If traffic is quiet — decrement k
- If k reaches zero — run `ipset del blocklist <ip>`, remove from table

## Libraries and Modules Involved

### Python Standard Library

**`socket`** — opens and reads from the Unix domain socket. Specifically `socket.AF_UNIX` and `socket.SOCK_STREAM`. No external library needed.

**`struct`** — unpacks raw bytes into Python floats. The format string `"7d"` unpacks 7 consecutive `f64` values (each `d` is a double — 8 bytes). You'll add a timestamp field too.

**`subprocess`** — runs `ipset` and `iptables` shell commands from Python. This is how you talk to the kernel blocking layer — you call `subprocess.run(["ipset", "add", "blocklist", ip])` rather than using a kernel API directly.

**`logging`** — structured logging of every classification decision, block event, and k-decay release. Essential for evaluation — you need a log to prove your F1 score, detection latency, and false positive rate during testing.

**`collections.defaultdict`** — clean way to manage the `blocked_ips` table with default values for `k_remaining` and `block_count`.

**`scikit-learn`** — `RandomForestClassifier` for training and `predict_proba()` for runtime classification. Also `joblib` (bundled with scikit-learn) for saving and loading the trained model to disk so you don't retrain on every startup.

**`numpy`** — assembles the 7 features into the array format `scikit-learn` expects. One line: `np.array([[h, r, mean_h, mean_r, sigma_h, sigma_r, proto_ratio]])`.

**`pandas`** — for the offline training phase only. Reads your labelled CSV of captured feature vectors, handles train/test split, feeds data to the RF trainer. Not needed at runtime.

## Stage 1 — Rust Statistical Sensor

### Job
Sit on the wire. Compute statistical features. Detect anomalies. Signal Stage 2.

### Capture
`pcap` opens interface in promiscuous mode with BPF filter `dst host <victim_ip>`. Inbound traffic only. Raw frames dropped into a bounded `crossbeam` channel immediately — the capture thread never stops to compute.

### Layer 1 — Per Packet
- EWMA updates its rate estimate: `value = alpha * new_rate + (1 - alpha) * value`
- Source IP incremented in `HashMap<IpAddr, u32>`
- Protocol counter incremented — TCP, UDP, or ICMP

### Layer 2 — Per Window (every 50th packet)
- EWMA read as snapshot → scalar `r`. EWMA never resets — carries memory across windows.
- Shannon entropy computed from HashMap → scalar `h`. HashMap then cleared.
- Protocol ratio computed from TCP/UDP/ICMP counts → scalar `proto_ratio`. Counts cleared.
- Packet counter resets to zero.

### Layer 3 — Per Window, Immediately After Layer 2
- `Welford(r).update(r)` → updates `mean_r`, `sigma_r` → checks `r > mean_r + 2*sigma_r`
- `Welford(h).update(h)` → updates `mean_h`, `sigma_h` → checks `h < mean_h - 2*sigma_h`
- If either breaches → serialize feature vector into `#[repr(C)]` struct → write to Unix socket

### Welford Note
Welford's `n` counts **windows**, not packets. After 50 packets, Welford has seen exactly one sample. After 500 packets, ten samples.

udp_count: u32                       ← clears every window

## Training Pipeline (Offline — Before Runtime)

1. Run Stage 1 fully on lab traffic — normal traffic first, then each attack type
2. Log every feature vector with a label: `0` = benign, `1` = attack
3. Load labelled CSV into `pandas`
4. Train `RandomForestClassifier` on the dataset
5. Evaluate on held-out test split — confirm F1 ≥ 0.98
6. Save model to disk via `joblib.dump(model, "model.pkl")`
7. Stage 2 loads `model.pkl` on startup — never retrains at runtime
